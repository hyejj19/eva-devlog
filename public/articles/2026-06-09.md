## 정규화란

정규화란 관계형 데이터베이스에서 데이터 중복을 줄이고, 이상 현상을 방지하기 위해 테이블 구조를 체계적으로 분해하는 과정이다. 하나의 테이블에 여러 주제의 정보가 섞여 있으면 같은 데이터가 반복 저장되거나, 특정 데이터를 수정할 때 여러 행을 함께 고쳐야 하는 문제가 생긴다.

정규화는 이런 문제를 줄이기 위해서 각 테이블이 하나의 명확한 의미를 가지도록 설계하는 방법이다.

## 왜 해야하나

정규화를 하지 않았을 경우 발생할 수 있는 대표적인 이상현상은 다음과 같다.

**삽입 이상**

새 데이터를 넣고 싶은데 불필요한 다른 데이터가 있으면 삽입할 수 없는 문제이다.

예를 들어 `학생_수강(학번, 이름, 과목코드, 과목명)` 테이블만 있다면, 학생 없이 새 과목만 따로 저장을 할 수 가 없다.

**삭제 이상**

특정 데이터를 삭제했을 때, 원하지 않는 다른 정보까지 함께 사라지는 문제이다.

예를 들어, 어떤 학생의 수강 정보를 삭제했는데 그 과목에 대한 정보까지 사라질 수 있다.

**수정 이상**

같은 과목명이 여러 행에 반복 저장되어 있을 때, 과목명이 변경되면 관련된 모든 행을 수정해야 한다.

일부 행만 수정되면, 같은 과목 코드인데 과목명이 서로 다른 데이터 불일치가 생길 수 있다.

## 단계

정규화는 여러 단계의 정규형을 만족하도록 테이블을 분해하는 방식으로 진행된다.

**제 1 정규형, 1NF**

제 1 정규형은 모든 속성 값이 원자값을 가져야 한다는 조건이다. 즉, 한 칸에 여러 값을 넣으면 안 된다.

나쁜 예:

좋은 예:

<table><thead><tr><th>학번</th><th>이름</th><th>과목</th></tr></thead><tbody><tr><td>1</td><td>김철수</td><td>DBMS</td></tr><tr><td>1</td><td>김철수</td><td>운영체제</td></tr></tbody></table>

**제 2 정규형, 2NF**

제 2 정규형은 1NF를 만족하면서, 기본 키의 일부에만 의존하는 속성을 제거한 상태이다.

주로 기본키가 복합키일 때 문제가 된다. 복합키를 기본키로 쓰는 테이블에서 각 컬럼은 기본키 전체에 의존해야 한다.

예를 들어, `수강(학번, 과목코드, 학생이름, 과목명, 성적)` 에서 기본키가 `(학번, 과목코드)` 라면 다음과 같은 종속성이 있다.

```plain text
학번 -> 학생이름
과목코드 -> 과목명
(학번, 과목코드) -> 성적
```

학생 이름은 학번에만 의존하고, 과목명은 과목 코드에만 의존한다. 따라서 부분 함수 종속이 존재하므로 2NF 를 만족하지 않는다.

이런 경우 만약 한 학생이 여러 과목을 수강하게 되거나, 한 과목을 여러 학생이 수강하게 될 경우 중복 저장되는 문제가 생길 수 있다.  그 결과 학생 정보나 과목명이 수정될 경우 관련된 모든 행을 수정해야 하며, 일부 행만 수정되면 데이터 불일치가 발생할 수 있다.

이를 분해하면 다음과 같다.

```plain text
학생(학번, 학생이름)
과목(과목코드, 과목명)
수강(학번, 과목코드, 성적)
```

**제 3 정규형, 3NF**

제 3 정규형은 2NF 를 만족하면서, 기본키가 아닌 속성이, 또 다른 기본키가 아닌 속성을 결정하면 안 된다. 이를 이행적 함수 종속을 제거한 상태라고 표현한다.

이행적 함수 종속이란,

```plain text
A -> B
B -> C
```

그래서 결과적으로,

```plain text
A -> C
```

가 된다.

`학생(학번, 학생이름, 학과코드, 학과명)` 과 같은 테이블이 있고 기본 키가 `학번` 이라면 함수적 종속성은 아래와 같다.

```plain text
학번 -> 학생이름, 학과코드
학과코드 -> 학과명
```

이 흐름을 뜯어보면,

```plain text
학번 -> 학과코드 -> 학과명
```

이런 구조를 이행적 종속이라고 한다.

이 구조를 3NF 정규화를 통해 분해하면 아래와 같다.

```plain text
학생(학번, 학생이름, 학과코드)
학과(학과코드, 학과명)
```
