## 들어가며.

자바스크립트는 싱글 스레드 언어다. 그런데 우리가 코드를 쓸 때 싱글 스레드라는 사실을 망각하고 각종 비동기를 자유롭게 사용할 수 있는 것은, 브라우저와 Node.js에 내장된 이벤트 루프 덕분이다. 또한 이벤트 루프에 의한 동작은 React 설계 방식에도 적용되어있다. 웹에서 찾은 실제 케이스를 바탕으로 이벤트 루프에 대해서 공부해보는 시간을 가져보자.

## 이벤트 루프의 기본 구조

### JS 엔진과 이벤트 루프는 별개다.

이벤트 루프는 브라우저 혹은 Node.js 환경이 제공하는 별개의 구조이다. 자바스크립트를 실행하는 V8 엔진은 코드를 실행하는 역할인 것이고, 이벤트 루프는 브라우저 (혹은 Node.js)를 통해 제공된다.

예를 들어, `fetch` api 를 호출하면, V8이 브라우저의 웹 API 에게 요청 처리를 넘기고, 브라우저의 네트워크 스레드가 실제 HTTP에 요청을 보내고 응답을 기다린다. 응답이 오면 브라우저가 이벤트 루프의 큐에 콜백을 넣어준다.

기다려야 하는 일은 브라우저에 위임하고, 메인 스레드를 블로킹 하지 않는 것. 이게 이벤트 루프의 기본 역할이다.

### 큐의 종류

큐의 종류에는 매크로태스크 큐와 마이크로태스크 큐가 있다. 

처음에는 태스크 큐(매크로태스크 큐) 하나만 있었다. `setTimeout`콜백, 클릭 이벤트 핸들러 등 독립된 작업 단위를 스케줄링 하기 위한 큐였고, 이벤트 루프는 이 큐에서 작업을 하나 꺼내 실행하고, 렌더링 기회를 주고, 다음 작업을 거내는 방식으로 동작했다.

문제는 ES2015에서 Promise 가 등장하면서 본격화되었다. `fetch` 응답이 도착하면, 브라우저가 매크로태스크를 스케줄링하여 Promise 를 resolve 하는데, 그 이후 `.then()` 으로 이어지는 후속처리는 독립 작업이 아닌 하나의 논리적 흐름으로, 매크로태스크큐의 작업 단위와 성격이 다르다. 

만약 이 후속 처리를 매크로태스크 큐에 넣는다면, 이미 큐에 쌓인 다른 콜백들 뒤로 밀려 Promise 체이닝이 느려지고, 비동기 흐름의 일관성이 깨질 수 있다. 그래서 이런 처리는 마이크로태스크 큐를 통해 처리된다.

마이크로태스크 큐의 설계 의도는, 논리적인 비동기 흐름이 중간에 다른 작업에 의해 끊기지 않도록 보장하는 것이다. 예를 들어, `fetch(url).then(res => res.json()).then(data => process(data))`와 같은 체인에서, 각 `.then()` 콜백이 매크로태스크 큐에 들어갔다면 중간에 setTimeout이나 클릭 핸들러가 끼어들 수 있다. 마이크로태스크 큐는 이 체인이 연속으로 실행되도록 보장한다.

### 실행 순서

매크로태태스크 하나 실행 → 마이크로태스크 큐 전부 비우기 → (브라우저가 필요하다면) 렌더링 → 다음 매크로태스크 실행

핵심 규칙:

- 매크로태스크는 한 사이클에 하나만 처리한다.
- 마이크로태스크는 큐가 빌 때까지 전부 처리한다. (도중에 추가된 것도 포함)
- 두 큐는 별개의 독립된 큐, 소속 관계가 아닌 실행 순서의 규칙으로 연결된다.

진짜로 그렇게 동작하는지 확인하기 위해 아래와 같은 함수를 호출해볼 수 있다.

```javascript
function callTasks(){
    console.log('스크립트 시작(매크로태스크)');

    setTimeout(()=>console.log('2'),0);
    setTimeout(()=>console.log('3'),0);

    Promise.resolve()
        .then(()=>console.log('4'));
        .then(()=>console.log('5'));
        .then(()=>console.log('6'));
    
    console.log('스크립트 끝');
}

/*
[호출 결과]
스크립트 시작(매크로태스크)
스크립트 끝
4 -> 5 -> 6 -> 2 -> 3
*/
```

스크립트가 실행되면서  setTimeout 으로 등록된 콜백 두 개가 매크로태스크 큐에 등록된다. 그리고 이미 resolve 가 완료된 then 내부 콜백이 마이크로 태스크 큐에 들어가게 된다.

스크립트가 종료된 이후 곧바로 모든 마이크로태스크 큐를 비우고(4,5,6) 그 다음 매크로태스크 큐를 순차 실행했기 때문에 2, 3이 뒤에 찍힌다.

### 렌더링 타이밍

브라우저의 메인 스레드는 JS 실행과 렌더링(스타일 계산, 레이아웃, 페인트)을 모두 담당한다. 하나의 스레드이므로 둘을 동시에 처리할 수 없고, 이벤트 루프가 이 순서를 관리한다.

WAHTWG HTML 스펙에 따르면, 매크로 → 마이크로 → 렌더링의 순서는 보장되지만, 렌더링이 매 사이클 일어난다는 보장은 없다. DOM 변경 여부, CSS 애니메이션 진행 여부, 마지막 렌더링 이후 시간, 탭의 가시성 등을 종합적으로 고려해 브라우저가 필요하다고 판단할 때에만 렌더링한다.

이런 구조 때문에, 마이크로태스크가 너무 오래 걸리면 렌더링이 밀릴 수 있다. 매크로태스크는 하나씩 처리되므로 사이클이 자주 돌고, 그래서 렌더링 판단의 기회가 자주 오지만, 마이크로태스크는 큐가 전부 비워질 때까지 사이클이 끝나지 않아 렌더링 단계에 도달하지 못하기 때문이다.


## Case 1: async/await 을 썼는데 서버가 멈추는 이유.

> 참고: [How even quick Node.js async functions can block the Event-Loop | Snyk](https://snyk.io/blog/nodejs-how-even-quick-async-functions-can-block-the-event-loop-starve-io/)

```javascript
app.get('/compute', async (req, res) => {
  const hash = crypto.createHash('sha256')
  for (let i = 0; i < 10_000_000; i++) {
    await asyncUpdate(hash)
  }
  res.send(hash.digest('hex'))
})

async function asyncUpdate(hash) {
  hash.update(randomString())  // 동기 연산. 비동기 작업이 없다.
}
```

위 코드에서 개발자의 의도는 `await` 을 넣어서 매 반복마다 다른 작업이 끼어들 수 있도록 양보하고자 했다. 그런데 실제로는 서버가 완전히 멈추고 다른 모든 요청이 타임아웃 되었다고 한다.

### 왜 이런 일이 생기는가?

`asyncUpdate` 는 `async` 함수이기 때문에 Promise 를 반환하지만, 내부에 비동기 작업이 없기 때문에 즉시 resolve 된다. `await` 은 `Promise.then()` 의 syntatic sugar 이고, 그렇기에 `await` 아래의 나머지 코드, 즉 for문의 다음 반복이 마이크로태스크큐에 들어가게 된다.

이벤트 루프는 앞서 살펴본 바와 같이 마이크로태스크 큐를 전부 비울 때까지 다음 단계로 넘어가지 않기 때문에, 천 만번의 반복이 마이크로태스크 큐 안에서만 계속 돌면서 다음 매크로태스크 단계에 절대 도달하지 못한다. 그러니까 await 을 사용하면 멈춘 척을 하는 것이고 실질적으로는 멈춰지지가 않는다.

### 해결: 매크로태스크 큐를 경유하여 강제 양보

위에서 살펴본 바와 같이, 만약 반복 중간에 다른 작업의 끼어들기를 가능하게 하고 싶다면, 각 작업을 매크로태스크에 담길 수 있게 우회할 수 있다. 예를들면 아래와 같이 변경하면, fake 함수가 실행하는 중간중간 예약된 매크로태스크인 `setTimeout`의 콜백이 실행되는 것을 볼 수 있다.

```javascript
async function fake() { 
	return new Promise(resolve => setTimeout(resolve, 0));
}

async function run() {
	for (let i = 0; i < 1000; i++){
		await fake();
		if(i % 100 === 0) console.log(`마이크로태스크 반복: ${i}`);	
	}
}

setTimeout(() => console.log('매크로태스크 실행'),0);
setTimeout(() => console.log('매크로태스크 실행'),1000);
setTimeout(() => console.log('매크로태스크 실행'),2000);
setTimeout(() => console.log('매크로태스크 실행'),3000);
run();
```

### setTimeout의 한계와 대안

그런데 위 방식에는 문제점이 있다. HTML 에 스펙에 의하면, `setTimeout(fn,0)` 은 5회 이상 중첩시 최소 4ms 의 지연이 강제된다. 그 이유는 타이머 남용으로 인한 성능 저하를 방지하기 위해서 애초에 설계가 이런 방식으로 되어있다. 

이를 회피하기 위한 대안으로는,

- `SetImmediate` : Node.js 전용이라서 웹에선 활용이 어렵지만. 4ms의 지연 없이 작업을 매크로태스크 큐에 넣는다.
- `MessageChannel` : 원래는 웹 워커 간 통신 용이지만, 같은 스레드에서 자기 자신에게 메시지를 보내면 핸들러가 매크로태스크로 실행된다.

React 18의 스케줄러가 이 MessageChannel 방식을 사용한다. Concurrent 렌더링을 통해 리액트는 이제 렌더링 phase 중간에도 다른 작업을 끼워 넣을 수 있게 되었는데, 이게 가능한 이유가 바로 5ms 마다 MessageChannel 을 사용해 나머지 작업을 매크로태스크 큐에 넣고 다른 작업에게 순서를 양보하기 때문이다.


## Case 2: React는 이벤트 루프를 어떻게 활용하는가 - 스케줄링 전략

> 참고1 : [Automatic batching for fewer renders in React 18 · reactwg Discussion #21](https://github.com/reactwg/react-18/discussions/21)

> 참고2: [How React Scheduler Works Internally — jser.dev](https://jser.dev/react/2022/03/16/how-react-scheduler-works/)

### React 17의 한계 

React 17 까지는 배칭(여러 setState를 모아서 한 번에 렌더링) 은 React 가 감싼 이벤트 핸들러 안에서만 작동했다.

```javascript
// React 17 — 이벤트 핸들러 안: 배칭 O
function handleClick() {
  setCount(1)    // 모아둠
  setFlag(true)  // 모아둠 → 끝나면 한 번에 리렌더링
}

// React 17 — .then() 안: 배칭 X
fetch('/api').then(res => {
  setCount(1)    // → 즉시 리렌더링
  setFlag(true)  // → 또 즉시 리렌더링
})
```

`.then()` 콜백 등은 React가 감싼게 아니라 브라우저의 마이크로태스크 큐에서 실행된 것이기 때문에, React17은 이를 setState 마다 즉시 렌더링 해다.

### 1단계: 자동 배칭 - 마이크로태스크로 렌더링 요청을 미룬다

React 18부터 `createRoot`를 사용하면, `setState` 호출시 위치에 상관없이 즉시 렌더링을 하지 않고, 업데이트를 내부 큐에 쌓은 뒤 렌더링이 필요하다는 신호를 마이크로태스크로 예약한다.

이 마이크로태스크의 역할은 렌더링을 직접 실행하는 것이 아니라, 스케줄러에게 렌더링을 요청하는 것이다. 스케줄러는 이 요청을 받아 `MessageChannel`을 통해 실제 렌더링을 매크로태스크로 등록한다.

```plain text
setState 호출 (여러 번)
  → 업데이트를 내부 큐에 모음
  → "렌더링 필요해"를 마이크로태스크로 예약

현재 코드 끝남

[마이크로태스크] 스케줄러에게 "렌더링 해줘" 전달
  → 스케줄러가 MessageChannel로 매크로태스크 등록

[매크로태스크] 실제 렌더링 실행
  → Render Phase (fiber tree 비교)
  → Commit Phase (DOM 변경)
```

### 2단계: 렌더링 실행 - 우선순위에 따라 방식이 달라짐

스케줄러가 매크로태스크로 등록한 렌더링이 실행될 때, 모든 렌더링이 같은 방식으로 실행되는 것은 아니다. React 18은 setState에 우선순위(Lane)을 부여하고, 이에 따라 렌더링 방식이 갈린다.

**동기 렌더링(SyncLane) : **클릭, input 등 사용자 인터랙션에 의한 업데이트는 높은 우선순위를 받는다. 그래서 Render Phase가 시작되면 끝날 때까지 쭉 실행한다. 사용자에 의한 이벤트 결과는 즉시 보여야 하기 때문이다.

**동시성 렌더링(TransitionLane) : **`startTransition` 이나 `useDeferredValue` 로 감싼 업데이트는 낮은 우선순위를 받는다. 이 경우 Render Phase 에서 fiber 노드를 하나씩 처리하면서, 5ms가 지나면 나머지를 `MessageChannel`을 통해 새 매크로태스크로 예약한다. 앞서 살펴본 바와 같이, 매크로태스크 사이에는 사용자 입력 등 다른 이벤트가 처리될 수 있기 때문에, 무거운 렌더링 작업을 하더라도 input이 끊기지 않을 수 있다.

```plain text
[매크로태스크] 렌더링 실행
  → SyncLane이면:
      Render Phase (끝까지) → Commit Phase → 완료

  → TransitionLane이면:
      [매크로태스크 1] Render Phase 시작 → 5ms 경과 → 나머지 재예약
      [브라우저] 사용자 입력 처리, 페인트 등
      [매크로태스크 2] 중단 지점부터 이어서 → 5ms 경과 → 재예약
      [매크로태스크 N] 순회 완료 → Commit Phase → DOM 반영
```

이것이 가능한 이유는 React16에서 도입된 Fiber 아키텍쳐에 있다. 이전의 Stack Reconciler 는 재귀 방식으로 트리를 순회했기 때문에, 진행 상태가 call stack 에 존재했고, 그렇기에 중간에 멈추면 어디까지 처리했는지를 알 수가 없었다.

Fiber 는 각 노드가 child, sibling, return 포인터를 갖고 있어서, 재귀 없이도 순회할 수 있다. 현재 작업중인 노드를 포인터로 기억하고 있기 때문에, 5ms 가 지나면 작업을 중단했다가, 나중에 다시 그 지점부터 이어서 처리를 할 수가 있다.

Render Phase가 완료되면 같은 매크로태스크 안에서 바로 Commit Phase로 이어진다. Commit Phase는 쪼개지 않고 동기적으로 한 번에 실행한다. (왜냐면 실제 DOM 을 바꾸는 작업이라서.)

### 정리

React는 이벤트 루프의 두 큐를 각각 다른 목적으로 활용한다.

- 마이크로태스크: 현재 코드가 끝날때까지 기다렸다가 렌더링을 요청하여 렌더링을 배치처리. 
- 매크로태스크: 렌더링 자체를 매크로태스크로 실행하되, 낮은 우선순위의 업데이트는 5ms 단위로 쪼개어 브라우저에 양보한다.

## Case 3: useEffect vs useLayoutEffect

> 참고1: [React 공식 문서: useLayoutEffect](https://react.dev/reference/react/useLayoutEffect)

> 참고2: [When do useEffect() callbacks get run? Before paint or after paint? — jser.dev](https://jser.dev/2023-08-09-effects-run-paint/)

React 공식문서의 예제를 참고해서 간단한 툴팁 컴포넌트를 작성했다. 이 컴포넌트는 useEffect 내부에서 ref 요소의 바운더리를 계산하여 포지션을 결정하도록 한다. 

```javascript
function TooltipWithEffect({targetRect, children}: TooltipProps) {
  const ref = useRef<HTMLDivElement>(null);
  const [tooltipHeight, setTooltipHeight] = useState(0);

  useEffect(() => {
    setTooltipHeight(ref.current!.getBoundingClientRect().height);
  }, []);

  let x = targetRect.left;
  let y = targetRect.top - tooltipHeight;
  if (y < 0) y = targetRect.bottom;

  return (
    <div
      style={{
        position: 'fixed',
        pointerEvents: 'none',
        left: 0,
        top: 0,
        transform: `translate3d(${x}px, ${y}px, 0)`,
      }}
    >
      <div
        ref={ref}
        className="bg-gray-800 text-white text-sm px-3 py-2 rounded shadow-lg border border-gray-600"
      >
        {children}
      </div>
    </div>
  );
}
```

이 코드의 문제점은, 아주 간헐적으로 위치가 깜빡이며 잘못 계산되었다가 원래 위치로 돌아온다는 것이다. 그래서 이런 케이스에는 `useLayoutEffect` 사용하라는 내용이 있다.

### 왜 다르게 동작할까?

이 문제는 Commit Phase 가 이벤트 루프 위에서 어떻게 실행되는지를 이해하면 이해하기 쉽다.

React의 Commit Phase 는 하나의 매크로태스크 안에서 동기적으로 실행된다. 내부 순서는 아래와 같다.

1. DOM 변경 적용 (동기)
1. `useLayoutEffect` 실행 (동기) - 동일한 태스크 안에서 실행됨.
1. `useEffect` 를 Scheduler 에 예약 - 별도의 매크로태스크로 등록된다.

`useLayoutEffet` 는 Commit Phase 와 같은 매크로태스크 안에서 동기적으로 실행되기 때문에, 이 코드가 끝나기 전에는 브라우저가 화면을 그릴 수 없다. 따라서 이 과정에서 DOM 을 측정 및 수정하는 작업을 진행하면 페인트 전에 완료되기 때문에 사용자는 잘못된 상태를 볼 틈이 없다. 

반면 `useEffect` 는 다른 매크로태스크로 분리된다. 앞서 이벤트루프와 브라우저 렌더링 과정에 대해 살펴본 바와 같이, 매크로태스크 큐의 실행 사이에 브라우저의 렌더링 기회가 있기 때문에, Commit phase 가 끝나고 useEffect 가 실행되기 전에 브라우저가 화면을 먼저 그릴 수 있다. 그렇기 때문에 잘못된 위치에 한 프레임 보이고 그 뒤에 원래 의도한 위치에 그려지게 될 가능성이 있다.

### useEffect는 브라우저 페인트 후에 실행 되는 것이 아니다.

여기서 주의할 점은, `useEffect` 가 항상 페인트 실행 이후에 실행 되도록 시점을 보장하는 것이 아니라, 대체적으로 effect가 실행되기 전 브라우저 페인트가 일어나도록 만든다는 것이다.

"React will **generally** let the browser paint the updated screen first before running your Effect." << 라고 [useEffect 문서](https://react.dev/reference/react/useEffect)에 나와있음.

그래서 위 케이스에서도 만약 페인트 이후에 실행된다고 보장이 된다면, 매번 깜빡이는 증상이 있어야 하는데 그렇지 않고 아주 가끔 그런 현상이 있는걸로 보여진다. 

아래와 같은 경우에는 useEffect도 페인트 전에 실행될 수 있기 때문에, 깜빡임이 발생하지 않는다.

- useLayoutEffect 안에서 state를 업데이트하면, React 는 새 렌더링 전에 pending 된 useEffect 를 먼저 flush 한다.
- 클릭 등 사용자 인터랙션으로 트리거된 effect 는 SyncLane으로 처리되어 페인트 전에 실행되 ㄹ수 있다.
- 작업량이 매우 적으면 스케줄러가 양보하지 않고 바로 실행한다.

그래서 이 점을 고려해 useEffect 와 useLayoutEffect의 설계 의도에 대한 차이점을 정리해보자면,


- `useLayoutEffect`: 페인트를 블로킹. 따라서 DOM 측정, 수정 작업이 사용자에게 보이기 전에 끝나야 할 때 사용.
- `useEffect`: 페인트를 블로킹하지 않도록 설계. 그래서 브라우저에게 먼저 그릴 기회를 양보할 수도 있다.

## Case 4: Promise.resolve().then(setState) — 업데이트 순서가 뒤바뀌는 버그

> 참고: [React Issue #24649](https://github.com/facebook/react/issues/24649)

이슈 리포터가 제시한 [원본 코드](https://codesandbox.io/p/sandbox/react-state-order-bug-o4izk6?file=%2Fsrc%2FApp.js%3A1%2C1-25%2C1)는 아래와 같다. 

```javascript
import React, { useState, useEffect } from "react";

export default function App() {
  var [submitCount, setSubmitCount] = useState(0);
  const [isSubmitScheduled, setIsSubmitScheduled] = useState(false);
  const scheduleSubmit = () => setIsSubmitScheduled(true);

  useEffect(() => {
    if (isSubmitScheduled) {
      setIsSubmitScheduled((prev) => !prev); // A
      Promise.resolve().then(() => {
        setSubmitCount((prev) => prev + 1); // B
      });
    }
  }, [isSubmitScheduled]);

  console.log("render", { submitCount, isSubmitScheduled });

  return (
    <button type="button" onClick={scheduleSubmit}>
      Hello
    </button>
  );
}
```

`useEffect` 내에서 state를 업데이트하는 두 가지 로직 A와 B가 있는데, 원래대로라면 순서대로 실행이 되어야 하니 A→B의 순서가 되어야 한다. 그런데 왜 B→A 의 순서로 동작하냐는 내용의 이슈이다.

[메인테이너의 답변](https://github.com/facebook/react/issues/24649#issuecomment-1442902967)은 아래와 같다.

> Interesting. The first state update is marked as DefaultLane because flushPassiveEffects sets the event priority; the second state update is outside that call stack so it reads the priority from window.event, which after a microtask is still the click event, leading to it getting treated as a discrete event and thus SyncLane.


즉 마이크로태스크큐에 의해 실행된 작업은, 리액트의 관리 컨택스트를 벗어나기 때문에 우선순위 오판의 문제가 발생한다는 설명이다.

React 18은 모든 setState에 우선순위(lane)을 부여한다. 사용자 인터랙션에 의한 업데이트는 높은 우선순위, useEffect 등에서의 업데이트는 낮은 우선순위. 

useEffect 콜백은 앞선 사례에서 살펴봤듯 React 가 관리하는 매크로태스크에서 실행된다. A가 호출될 때에 React 는 useEffect 컨택스트임을 알고있기 때문에 DefaultLane(낮은 우선순위)를 부여한다.

B는 `Promise.resolve.then()` 즉 앞서 살펴본 바와 같이 마이크로태스크로 관리된다. React 입장에서는 자신의 관리 밖에서 호출된 것이기 때문에 `window.event` 를 읽어 우선순위를 추론하게 된다. 

사용자의 클릭으로 인해 컴포넌트가 마운트 된 경우에는, 이 시점에 `window.event` 에 아직 클릭 이벤트가 남아있을 수 있기 때문에, React 입장에서는 이를 SyncLane(높은 우선순위)로 오판할 가능성이 있다.

## 마무리

위 4가지 케이스를 관통하는 핵심은, 이벤트 루프는 브라우저(혹은 Node.js 런타임)가 제공하는 인프라이고, React 든 뭐든 그 위에서 돌아간다는 것이다.

- case1: 마이크로태스크큐가 비지 않으면 렌더링이 밀릴 수 있음
- case2: React 18은 마이크로태스크로 배칭, 매크로태스크로 양보
- case3: useLayoutEffect는 Commit Phase와 같은 매크로태스크에서 동기 실행되고, useEffect는 별도 매크로태스크로 분리된다
- case4: 이벤트루프 타이밍과 React 내부 스케줄러 사이에 미묘한 틈이 있을 수 있다.

이벤트 루프의 규칙은, 마이크로태스크는 연속성을 보장하고, 매크로태스크는 작업 사이 다른 작업의 양보 기회를 제공한다는 것을 알면 위 케이스에 대한 이해가 더 쉬울 것이다.
