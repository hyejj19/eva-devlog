## 인덱스란?

인덱스는 데이터베이스에서 데이터를 빠르게 찾기 위해 사용하는 보조 자료도구이다.

책에서 원하는 내용을 찾을 때 처음부터 끝까지 읽지 않고 목차나 색인을 볼 수 있는 것처럼, DBMS도 테이블 전체를 처음부터 끝까지 탐색하지 않기 위해 인덱스를 사용한다.

인덱스는 보통 다음과 같은 형태로 생각할 수 있다.

```plain text
검색키 값 → 실제 데이터 위치
```

여기서 검색키는 검색할 때 기준이 되는 컬럼이다.

예를 들어 다음 쿼리에서는 email 이 검색키가 된다.

```sql
SELECT *
FROM users
WHERE email = 'a@test.com';
```

검색키는 꼭 기본키일 필요는 없다. `id`, `email`, `name`, `created_at`처럼 조회 조건에 자주 사용되는 컬럼이면 인덱스의 검색 키가 될 수 있다.

## 인덱스가 필요한 이유

인덱스가 없으면 DBMS는 원하는 데이터를 찾기 위해 테이블을 처음부터 끝까지 확인해야 하 ㄹ수 있다. 이것을 Full Table Scan 이라고 한다. 예를 들어 회원이 1,000만 명 있는 테이블에서 특정 이메일을 찾는다고 가정하자.

```sql
SELECT *
FROM users
WHERE email = 'test@example.com'
```

인덱스가 없다면 DBMS 는 많은 행을 하나씩 비교해야 한다.

하지만 `email` 컬럼에 인덱스가 있다면, 인덱스를 먼저 확인해서 해당 데이터가 있는 위치로 빠르게 이동할 수 있다.

즉 인덱스의 목적은 단순하다.

전체를 다 뒤지지 말고, 정리된 자료구조를 통해 빠르게 찾아가자!

## 인덱스의 장점과 단점

앞서 살펴본 바와 같이, 인덱스의 가장 큰 장점은 조회 성능을 높일 수 있다는 것이다.

특히 `WHERE`, `JOIN`, `ORDER BY`, `GROUP BY`처럼 데이터를 찾거나, 연결하거나, 정렬하거나, 묶는 작업에서 도움이 된다. 인덱스는 검색키 값과 실제 데이터 위치를 정렬된 자료구조로 관리하기 때문에, DBMS가 테이블 전체를 훑지 않고 필요한 데이터에 빠르게 접근할 수 있게 해준다.

예를 들어 `WHERE` 조건에서는 원하는 값을 빠르게 찾을 수 있고, `JOIN`에서는 연결할 상대 행을 빠르게 찾을 수 있다. 또한 `ORDER BY`와 `GROUP BY`에서는 인덱스가 가진 정렬 순서를 활용해 별도의 정렬이나 그룹화 비용을 줄일 수 있다.

다만 인덱스가 있다고 해서 항상 사용되는 것은 아니다. DBMS 의 옵티마이저가 인덱스를 쓰는 거싱 더 효율적이라고 판단할 때 사용된다.

하지만 인덱스는 공짜가 아니다. 인덱스도 별도의 자료구조이기 때문에 저장 공간을 사용한다.

또한 데이터를 추가하거나 수정, 삭제할 때 테이블 뿐만 아니라 인덱스도 함께 관리해야 한다.

```sql
INSERT -> 인덱스에도 새 항목 추가
UPDATE -> 기존 인덱스 값 수정
DELETE -> 인덱스에서도 삭제 처리
```

그래서 인덱스를 너무 많이 만들면 조회는 빨라질 수 있지만, INSTER, UPDATE, DELETE 성능은 떨어질 수 있다.


## 주요 인덱스 종류

### 밀집 인덱스

모든 검색키 값에 대해 인덱스 트리를 가지는 방식이다.

```plain text
검색키 1 -> 데이터 위치
검색키 2 -> 데이터 위치
검색키 3 -> 데이터 위치
```

찾기는 빠르지만, 모든 값마다 인덱스 항목이 필요하므로 저장 공간을 많이 사용한다.

### 희소 인덱스

일부 검색키 값에 대해서만 인덱스 엔트리를 가지는 방식이다.

예를 들어 데이터가 학번 순서로 정렬되어 있다면, 모든 학생의 학번을 인덱스에 넣지 않고 각 페이지의 첫 번째 학번만 인덱스에 둘 수 있다.

```plain text
1000번대 시작 위치
2000번대 시작 위치
3000번대 시작 위치
```

공간은 적게 사용하지만, 보통 데이터가 검색키 순서로 정렬되어 있을 때 유용하다.

### 다단계 인덱스

인덱스 데이터가 많아지면 크기가 커진다. 그러면 인덱스 자체를 찾는 데도 시간이 걸릴 수 있다.

이때 인덱스 위에 또 다른 인덱스를 두는 방식이 다단계 인덱스다.

```plain text
상위 인덱스
   -> 하위 인덱스
      -> 실제 데이터 위치
```

책으로 비유하면 목차 안에 큰 목차, 중간 목차, 세부 목차가 있는 구조와 비슷하다.

뒤에서 살펴볼 B+Tree 인덱스는 이런 다단계 구조를 트리 형태로 정리한 대표적인 예라고 볼 수 있다.

### 해시 인덱스

검색키 값을 해시 함수에 넣어 버킷 주소를 계산하는 방식이다.

```plain text
검색키 값 -> 해시 함수 -> 버킷 주소
```

예를 들어 `user_id = 10`을 찾는다면, 10을 해시 함수에 넣고 나온 위치로 바로 이동한다.

해시 인덱스는 = 조건에서는 빠르다.

```sql
WHERE user_id = 10
```

하지만 범위 검색에는 적합하지 않다.

```sql
WHERE user_id > 10
WHERE created_at BETWEEN '2026-01-01' AND '2026-01-31'
```

해시는 값의 순서를 보존하지 않기 때문에, "10보다 큰 값들"처럼 순서가 필요한 검색에는 약하다.

### B+Tree 인덱스

DBMS 에서 가장 많이 사용되는 트리 계열 인덱스다.

B+Tree는 검색키를 정렬된 상태로 저장하므로, 트리를 내려가며 원하는 값이 있을 리프 노드를 빠르게 찾을 수 있다. 또한 범위 검색에서는 시작 키를 찾은 뒤 리프 노드를 순서대로 읽다가 범위를 벗어나면 탐색을 멈출 수 있다.

구조는 대략 다음과 같다.

```plain text
        루트 노드
       /    |    \
   중간 노드  중간 노드
    /   \       /   \
 리프 노드 -> 리프 노드 -> 리프 노드
```

B+Tree 에서 실제 검색키와 데이터 위치는 보통 리프 노드에 저장된다.

루트 노드와 중간 노드는 원하는 리프 노드까지 찾아가기 위한 안내판 역할을 한다.

예를 들어 `id = 70` 인 데이터를 찾는다면 DBMS 는 다음처럼 이동한다.

```plain text
루트 노드에서 방향 결정
-> 중간 노드에서 다시 방향 결정
-> 리프 노드 도착
-> id = 70의 데이터 위치 확인
```

즉, 테이블 전체를 처음부터 끝까지 읽는 것이 아니라, 트리를 따라 몇 개의 인덱스 페이지를 거쳐 원하는 위치 근처로 바로 이동한다.

B+Tree 가 범위 검색에 강한 이유는 리프 노드들이 검색키 순서대로 정렬되어 있고, 서로 연결되어 있기 때문이다.

예를 들어 다음 쿼리가 있다고 하자.

```sql
SELECT *
FROM users
WHERE age BETWEEN 30 AND 50;
```

DBMS 는 먼저 트리를 따라 `age = 30`이 있을만한 리프 노드를 찾는다.

리프 노드들이 옆으로 연결되어 있으므로, 다음 리프노드로 이동하면서 순서대로 읽으면 된다.

```plain text
[20, 25, 30] -> [35, 40, 45] -> [50, 55, 60]
          시작               계속 읽음        50까지 읽고 종료
```

따라서 B+Tree의 범위 검색은 다음 두 단계로 이루어진다.

1. 트리 탐색으로 범위의 시작 위치를 빠르게 찾는다.
1. 리프 노드를 옆으로 따라가며 범위가 끝날 때까지 순서대로 읽는다.
이 방식 덕분에 B+Tree 는 트정 값 하나를 찾는 검색뿐 아니라, 일정 구간을 조회하는 범위 검색에도 효율적이다.

다만 인덱스를 사용한다고 해서 항상 디스크 접근이 없는 것은 아니다. 인덱스 페이지도 필요하면 디스크에서 읽어와야 하고, 리프 노드에서 찾은 데이터 위치를 따라 실제 데이터 페이지를 추가로 읽어야 할 수도 있다. 그래도 전체 테이블을 읽는 것보다 훨씬 적은 수의 페이지를 읽게 되는 경우가 많기 대문에 조회 성능이 좋아진다.
