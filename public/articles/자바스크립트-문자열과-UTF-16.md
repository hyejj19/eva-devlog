`What if the inputs contain Unicode characters? How would you adapt your solution to such a case?`

## 유니코드란

- 유니코드 (Unicode) 는 전 세계의 모든 문자가 가진 고유 코드를 의미함.
- 과거에는 언어를 저장하는 규칙이 다 달라서 문자가 깨졌는데, 유니코드 즉 고유한 코드를 각 문자별로 저장하면서 규격화를 한 것. ex) A = U+0041, 가 = U+AC00, `🍎` = U+1F34E.
## follow up 에 대한 답변.

기존 문제의 제약 조건에는  `s and t consist of lowercase English letters.` 라는 단서가 붙었기 때문에, 어떤 접근 방식을 사용해도 괜찮았으나, 만약 유니코드를 포함한다면, 전 세계의 모든 문자가 인풋으로 들어오게 될 수도 있다는 것을 의미함.

자바스크립트는 내부적으로 UTF-16 방식으로 문자열을 처리하는데, 이모지나 일부 특수 유니코드의 경우에는 이 범위를 벗어나는 서로게이트 쌍이 될 수 있기 때문에 접근 방식을 달리 해야한다.

```typescript
// index 기반 접근
const str = "A🍎B";

console.log(str.length); // 4 (이모지를 1개가 아닌 2개로 쪼개서 본다.)

console.log(str[0]); // "A"
console.log(str[1]); // "�"
console.log(str[2]); // "�"
console.log(str[3]); // "B"

// iterator 기반 접근 (ES6 문법 부터 이에 대한 대응이 가능하다.)
const str = "A🍎B";
for (const char of str) {
	console.log(char);
}

// "A"
// "🍎"
// "B"
```

## UTF 포맷

- UTF 란, `Unicode Transformation Format`
- 문자별 유니코드를 실제 메모리에 저장하기 위한 인코딩 방식을 의미한다.
- 그래서 숫자 8은 8비트를, 16은 16비트를 기본 단위로 사용하여 인코딩 하겠다는 것을 의미한다.
- 웹에서는 UTF-8 포맷을 표준으로 사용하고, 자바스크립트는 UTF-16 포맷을 사용한다.
## UTF-16과 자바스크립트

- 초창기 유니코드 위원회는 16비트 내로 당시 6만 5천개 가량의 전세계 모든 문자를 처리할 수 있다고 보았다. 자바스크립트도 이 방식을 채택했고 이 때 당시의 16비트 고정 처리방식을 `UCS-2` 라고 한다.
- 고대 한자와 상형 문자, 이모지 등이 포함되면서 문자의 갯수가 10만개를 넘어 더이상 16비트로 표현할 수 없는 한계가 왔음. 그래서 이런 문자열에 대해 16비트를 두 개 연달아 총 32비3트를 하나의 문자로 보는 서로게이트 쌍이 생겨났고, 이 포맷을 UTF-16 이라고 함.
## UTF-8과 웹

- 웹 즉 네트워크 환경에서는 조금이라도 용량을 효율적으로 처리하는 것이 통신 속도에 중요한 영향을 미친다. 이를 위해 개발된 것이 UTF-8 포맷.
- 이 방식은 글자의 종류(유니코드 번호의 크기) 에 따라서 1~4 바이트까지 유동적으로 크기를 조절하는 가변길이 인코딩 방식이다.
- 한글 같은 경우는 3바이트 영역에 해당되어서 2바이트 고정 포맷인 utf-16에 비해 오히려 비효율적이지만, 웹 리소스는 대부분 영어 기준이기 때문에 UTF-8 포맷을 활용했을 때 더 효율적이다.
## 정리: 왜 인덱스 접근은 깨지고, 이터레이터는 안 깨지는가

- 웹에서는 효율적인 전송을 위해 UTF-8 인코딩을 표준으로 사용하지만, 자바스크립트 엔진은 역사적인 이유로 내부적으로 UTF-16을 사용한다. 브라우저가 리소스를 수신하면 UTF-8 바이트를 유니코드 코드 포인트로 디코딩하고, JS 엔진이 이를 다시 UTF-16 code unit 시퀀스로 변환하여 저장한다.
- 이 UTF-16 체계에서 이모지처럼 16비트 범위를 넘는 문자는 서로게이트 쌍, 즉 두 개의 code unit으로 쪼개져서 저장된다. `str[i]`는 이 code unit 단위로 접근하기 때문에 서로게이트 쌍의 절반만 반환하여 문자가 깨진다.
- 반면 ES6에서 도입된 `for...of` 이터레이터는 현재 위치의 code unit이 서로게이트 쌍의 앞부분인지를 확인하고, 맞다면 두 code unit을 합쳐서 하나의 완전한 문자로 반환한다. 이것이 인덱스 접근과 이터레이터 접근의 동작 차이가 발생하는 근본적인 이유다.
- 따라서 유니코드 문자를 안전하게 다루려면 인덱스 기반 접근 대신 `for...of`, 스프레드 연산자(`[...str]`), `Array.from(str)` 등 이터레이터 기반 접근을 사용해야 한다.
